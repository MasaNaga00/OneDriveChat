# Copilot 定型質問プロンプト集（案件エージェント用）

案件フォルダ全体を対象に、決まった切り口でまとめ・提案させるためのプロンプト集。
M365 Copilot の「定型質問（プロンプト）」として登録して使う想定。

**この案件ボットの想定：チーム内で共有 / 状況把握と次の打ち手が主用途 / メール等の
やり取り履歴を多く含む。** この前提に最適化している。

---

## 使い方

1. 案件ごとのエージェント（その案件フォルダを参照するエージェント）を開く
2. 下記のコードブロック内をそのまま定型質問として登録、またはチャットに貼り付け
3. 案件資料全体を踏まえた定型レポートが返る

各プロンプトは独立して使える。

### 登録の優先度（多すぎると使われなくなるため二層に分ける）

- **毎回使う主力（①②）** … 状況把握と次の打ち手。最初に登録すべき2つ。
- **必要なときに使う補助（③④⑤）** … 経緯整理・抜け漏れ・キャッチアップ。

---

# 主力プロンプト

## ① 進捗状況まとめ

```
あなたはこの案件の進行管理担当です。
この案件に関する資料全体を読み、現在までの進捗状況を以下の構成でまとめてください。
チームで共有するため、第三者が読んでも分かるように記述してください。

# 進捗サマリー
- 完了済みの事項（それぞれ根拠とした出典ファイル名を併記）
- 進行中の事項（現在の状態と、分かれば次の期限・担当者）
- 未着手・保留の事項

# 直近の動き
時系列で、最近の主要な出来事を3〜5点。日付と関係者が分かれば明記してください。

# 補足
資料から読み取れない点、または資料間で矛盾している点があれば指摘してください。

【ルール】
- 資料に書かれている事実だけを根拠にすること
- 「誰が・いつ・何を」をできるだけ明示すること（チーム共有のため）
- 推測が必要な場合は文頭に「推測：」と明記すること
- 各項目の末尾に、根拠とした出典ファイル名を必ず記載すること
- 資料にない情報を補わないこと
```

## ② 次にすべきアクションの提案

```
あなたはこの案件のプロジェクトマネージャーです。
この案件に関する資料全体を踏まえ、次に取るべきアクションを提案してください。

# 次のアクション提案
優先度の高い順に最大5件。各アクションについて以下を記載：
- 何を：具体的な作業内容
- なぜ：資料のどの状況がそれを必要としているか（根拠の出典ファイル名）
- 誰が・いつまでに：資料から分かる範囲で（不明なら「要確認」）

# リスク・注意点
放置すると問題になりそうな点を挙げてください。

【ルール】
- 資料の事実に基づくこと。資料にない前提を作らないこと
- 推測が必要な場合は「要確認」と明記すること
- 各提案の根拠に出典ファイル名を添えること
```

# 補助プロンプト

## ③ 経緯・決定事項の整理（メール履歴が多い案件向け）

やり取りの履歴が多いと、何がいつ決まったかが埋もれる。これを時系列で掘り起こす。

```
この案件のやり取り（メール・議事録など）を中心に資料を読み、
これまでの経緯と決定事項を整理してください。

# 決定事項の一覧
これまでに決まったことを時系列で。各項目に「いつ・誰が決めたか・出典ファイル名」を併記。

# 未決事項・宿題
まだ決まっていない、または誰かの対応待ちになっている項目。
分かる範囲で「ボールを持っている人（担当）」を明記してください。

# やり取りの流れ
主要なやり取りの流れを時系列で簡潔に。

【ルール】
- メールの送受信者・日付を手がかりに「誰が・いつ」を特定すること
- 決定事項と、単なる提案・検討中の区別をつけること
- 各項目に出典ファイル名を添えること
- 不明な点は「資料から特定できず」と明記すること
```

## ④ 案件キャッチアップ（新規参加・引き継ぎ向け）

```
この案件に初めて関わる人向けに、案件の概要を説明してください。

# この案件の概要
- 目的・ゴール
- 現在のフェーズ・状況
- 主な関係者と役割（資料から分かる範囲）
- 押さえておくべき重要事項 3〜5点
- 経緯の要点（これまでの大きな流れ）

各項目に出典ファイル名を添えてください。資料にない情報は補わないこと。
```

## ⑤ 抜け漏れチェック

```
この案件の資料を確認し、対応漏れや要注意の項目を洗い出してください。

# 要対応リスト
- 未回答・対応中のまま止まっている項目（特にメールでの依頼・問い合わせ）
- 期限が近い、または過ぎている可能性のある項目
- 決まっていない・保留中の事項

各項目に根拠の出典ファイル名と、分かれば担当者を添えてください。
推測が含まれる場合は「要確認」と明記してください。
```

# ピンポイント確認（定型登録ではなく手入力用）

特定の事項を都度調べるとき用。穴埋めして使う。

```
この案件の資料に基づいて、次の点について教えてください：「（知りたいことを記入）」

【ルール】
- 資料に書かれていることだけを答える
- 該当する記述がなければ「資料内に記載なし」と答える
- 回答の根拠となった出典ファイル名を明記する
```

---

# 精度を上げるコツ

- **役割を与える**（「進行管理担当」「PM」）→ 視点が定まり語り口が安定する
- **出力構造を見出しで固定** → 毎回同じ書式で返るので案件横断で比較しやすい
- **「誰が・いつ」を明示させる** → チーム共有では特に重要。メール履歴が多いほど効く
- **出典ファイル名を必ず書かせる** → 前処理で各資料に埋め込んだ出典表記を Copilot が拾う
- **「資料にない情報を補うな」と明示** → 事実に基づかない作文（ハルシネーション）を抑える
- **決定事項と検討中を区別させる** → 履歴が多い案件で「決まったこと」を誤認しないため

---

# 注意点（Copilot の制約とメール履歴の扱い）

- 案件フォルダ内のファイル数が多い場合、Copilot が一度に参照できる範囲に上限がある。
  ファイルが数十を超える案件では、全資料を完全には網羅できない可能性がある。
- **メール履歴が多い案件では特に注意**：古いやり取りが埋もれやすい。前処理（msg変換）で
  件名・日付・送受信者をメタ情報として頭に出しているため、これが手がかりになる。
  それでも漏れる場合は、③で時系列整理させてから②で打ち手を考える、と段階を分けると安定する。
- 対策：
  - 「特に〇〇.md と △△.md を重視して」と参照を誘導する一文を加える
  - 重要なメールだけ別途まとめた資料を1つ作っておくと、Copilot が拾いやすい
